
### 1. Definition
**Cardinality** refers to the number of **unique values** contained within a specific column of a dataset.

*   **High Cardinality:** Values are highly unique (e.g., `Transaction_ID`, `UUID`, `Email_Address`, `DateTime`).
*   **Low Cardinality:** Values repeat frequently (e.g., `Gender`, `Country`, `Status`, `Boolean`).

### 2. The Power BI Engine ([[Vertipaq|VertiPaq]]) Impact

Power BI uses an in-memory columnar engine ([[Vertipaq|VertiPaq]]) that relies heavily on **Dictionary Encoding** and compression. Cardinality is the single biggest factor determining your model's file size (RAM usage) and performance.

#### How it works:

1.  **The Dictionary:** The engine scans a column and creates a distinct list of values.
2.  **The Map:** It replaces the actual data in the rows with integer pointers referencing that dictionary.

#### The Cost:

*   **Low Cardinality (Good):**
    *   *Column:* `Status` ("Open", "Closed") across 10 million rows.
    *   *Storage:* The dictionary size is negligible (2 items). The data is stored as bits pointing to 0 or 1.
    *   *Result:* Extremely high compression ratio. Fast filtering.
*   **High Cardinality (Bad):**
    *   *Column:* `Transaction_UUID` (10 million unique strings).
    *   *Storage:* The dictionary must store 10 million unique strings. The pointers must be large enough to handle 10 million variations.
    *   *Result:* Almost no compression. The dictionary may be larger than the data itself. Slower load times.

### 3. Data Engineering & Modeling Implications

#### A. The "Split" Technique (Date/Time)
The most common high-cardinality killer is a `DateTime` column (e.g., `2023-01-01 14:05:01`).
*   **Problem:** 1 year of data down to the second = 31 million unique values.
*   **Solution:** Split this into two columns in the ETL/[[SQL]] stage.
    1.  `Date` (Cardinality: 365)
    2.  `Time` (Cardinality: 86,400)
    *   *Impact:* Reduces dictionary size significantly while keeping the same information.

#### B. Primary Keys & [[Relationships]]
While High Cardinality is bad for *grouping* visuals, it is **required** for the "One" side of a relationship.
*   **Best Practice:** Use Integer keys (Surrogate Keys).
*   **Avoid:** Using GUIDs or Strings as keys. A GUID is 128-bit random text; it cannot be compressed or sorted efficiently. Power BI has to create a massive Hash Dictionary to manage it.
*   **Engineering Step:** In your Data Warehouse, generate an Auto-Increment `INT` or `BIGINT` to serve as the PK, even if the source system uses GUIDs.

#### C. Granularity (Grain)
Cardinality is directly tied to the grain of the table.
*   **Data Engineering Question:** Does the business need every single transaction row, or do they need a daily summary?
*   **Aggregation:** Grouping data by `Day` and `Store` in [[SQL]] before loading to Power BI reduces row count and usually lowers the cardinality of high-variance columns.

### 4. Related Concepts

| Concept | Definition & [[Context]] |
| :--- | :--- |
| **Selectivity** | *[[SQL]] [[Context]].* How effectively a query filters data. High cardinality usually means high selectivity (good for SQL Index seeks). High cardinality is generally good for SQL retrieval but bad for Power BI storage. |
| **Auto-Exist** | *Power BI [[Context]].* A [[DAX]] behavior where the engine optimizes filters based on existing combinations of columns. High cardinality columns in the same table can cause performance issues here if used together in complex measures. |
| **Surrogate Key** | An artificial key (usually Integer `1, 2, 3...`) generated by the data engineering pipeline to replace natural, high-cardinality keys (like composite text keys). Essential for Star Schemas. |

### 5. Best Practices Checklist

1.  **Remove High Cardinality Columns:** If a column has high cardinality (like `Description` or `Log_ID`) and is not used for specific filtering or joining, **remove it** in Power Query.
2.  **Disable "Auto Date/Time":** In Power BI Options. This feature automatically creates a hidden table for every date column, bloating model size.
3.  **Optimize Text:** If you have a high cardinality text column you must keep, ensure it is trimmed and uppercase/lowercase standardized. "Apple" and "apple" are two distinct values in a dictionary.
4.  **Integers over Strings:** Always convert IDs to Whole Numbers if possible.
